# fix-brief — 문제 정리 포맷

진단/개선/리팩터/이슈 분석 결과를 **항상 아래 5단 구조**로 정리한다.
추측을 적지 말 것 — 각 항목은 실제 코드·로그·파일을 확인한 근거 위에서만 쓴다.

## 출력 규칙

1. **우선순위 정렬**: 항목들을 `기대효과 대비 드는시간`이 좋은 순(= 가성비 순)으로 나열한다.
   맨 위가 "지금 당장 할 것", 아래로 갈수록 "크고 위험한 것".
2. **각 항목은 정확히 이 5줄**을 가진다:

   | 필드 | 내용 |
   |---|---|
   | **문제** | 무엇이 비효율/결함인지 한 줄. 관측 가능한 사실만. |
   | **원인** | 왜 그렇게 됐는지. 파일·라인·설정 등 구체적 근거. |
   | **해결법** | 무엇을 어떻게 바꿀지. 실행 가능한 수준으로 구체적. |
   | **드는시간** | 소요 추정 + 위험도. (예: `5분·무위험`, `30분·중위험(임포트 깨질 수 있음)`) |
   | **기대효과** | 고치면 무엇이 좋아지는지. 가능하면 정량. |

3. 표 또는 소제목 어느 쪽이든 좋으나 **5개 필드 순서는 고정**(문제→원인→해결법→드는시간→기대효과).
4. 항목마다 **위험도**(무위험/저/중/고)와 **되돌림 가능 여부**를 드는시간 칸에 함께 표기 —
   사용자가 "이건 해줘 / 이건 보류" 판단을 바로 할 수 있게.
5. 표 뒤에 **한 줄 추천**: "지금 바로 칠 것 N개 / 결정 필요 M개".
6. 자가검증(테스트·빌드) 가능한 안전 항목은, 사용자가 허용하는 맥락이면
   포맷 제시 후 바로 적용하고 결과를 같은 5단으로 회신한다.

## 예시 (1개 항목)

**① CI가 전체 테스트를 안 돌림**
- **문제**: 860개 테스트 중 CI는 44개(`test_monitor.py`)만 실행 → 신규 하드닝 테스트가 게이트 밖.
- **원인**: `.github/workflows/test.yml`이 `pytest tests/test_monitor.py`로 단일 파일 고정.
- **해결법**: 해당 스텝을 `pytest`(전체)로 변경. 의존성은 이미 `requirements.txt` 설치됨.
- **드는시간**: 5분 · 무위험 · 되돌림 쉬움(1줄).
- **기대효과**: 회귀 방어 범위 44→860. 깨진 코드가 머지되는 걸 CI가 막음.

## 언제 쓰나

- 사용자가 코드/레포/워크플로의 문제점·개선안을 물을 때
- 여러 수정 후보를 우선순위로 제시해야 할 때
- 리팩터·정리·감사 결과를 보고할 때
