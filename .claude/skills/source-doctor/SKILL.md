# source-doctor — ❌ 비활성 소스 원인진단·처방

## 목적
`D:\mail\sites.json`에서 `enabled: false`인 소스(2026-07 기준 192개)를 대상으로
**① 원인 ② 해결방안 ③ 예상 토큰 ④ 기대효과**를 산출해 사용자가 "뭘 먼저 살릴지" 결정하게 한다.

## 절차 (읽기 전용 — sites.json·코드 수정은 별도 승인 후)

1. **대상 수집**: `sites.json`에서 `enabled:false` 목록 로드. 사용자가 소스명을 지정했으면 그것만.
2. **원인 클러스터링** (증거 우선순위 순):
   - `note` 필드 (수동 기록된 사유)
   - git 이력: `git log --oneline -- sites.json` + RESUME.md·SESSION_RECAP.md 의 비활성 사유 기록
     (예: 신규 등록 후 셀렉터 보강 pass2 대기, 404, SSL, JS렌더링 필요, 통합포털 중복)
   - `monitor.py`의 해당 type(html_table 등) 처리 요건 — 활성화에 무엇이 필요한지
   - **샘플 실접속 프로브**: 클러스터별 대표 2~3개만 requests(timeout 5s)로 상태코드·인코딩·JS필요 확인
     (192개 전수 접속 금지 — 느리고 차단 위험)
3. **클러스터별 처방표 작성** (개별 소스 나열은 대표 예시만, 표는 클러스터 단위):

   | 클러스터(원인) | 소스 수 | 해결방안 | 예상 토큰 | 기대효과 |
   |---|---|---|---|---|

   - **예상 토큰**: 셀렉터 1개 보강 ≈ 5~15k / 클러스터 일괄(스크립트) ≈ 30~80k / JS렌더링(playwright) ≈ 50~120k
   - **기대효과**: 해당 소스들의 예상 공고 수집량·평생목표(누락 0, recall 1순위) 기여도로 서술
4. **출력은 채팅 표로** (.md 파일 생성 금지 — 전역 규칙). 마지막에 "어느 클러스터부터 살릴까요?" 선택지 제시.
5. 사용자가 특정 클러스터 수정을 승인하면 → 셀렉터 보강은 기존 절차(RESUME의 pass2 절차·격리 워크트리) 재사용.

## 실제 복구 절차 (2026-07-23 검증됨 — 신규 스크립트 만들지 말고 이 순서대로)

기존 스크립트 4종이 이미 있다. **절대 새로 만들지 마라.**

1. `python -X utf8 scripts/diagnose_site_parse.py --all` — 전수 재진단(약 4분, `reports/site_parse_diag_*.json` 생성)
2. 직전 진단 파일과 대조해 ①뒤집힘↑(살릴 수 있는 것) ②뒤집힘↓(**운영 중인데 0건 = 회귀, 최우선**)을 뽑는다
3. `python -X utf8 scripts/autofill_selectors.py` (드라이런) → `--write` (셀렉터 주입)
4. **⚠️ 필수 검증 — 이걸 빠뜨리면 오히려 망가진다:** `--write`는 **사람이 배선해 둔 구체적 셀렉터까지 휴리스틱 값으로 덮어쓴다**.
   실행 후 반드시 백업본과 대조해, 소스마다 **옛 셀렉터 vs 새 셀렉터의 "유효행 수"**(`<a>` 있고 날짜 정규식 매치되는 행)를 실측하고
   옛것이 같거나 우세하면 **원복**하라. (2026-07-23 실측: 14건 변경 중 8건이 원복 대상 — K-SURE는 12건→0건으로 악화될 뻔)
5. 살아난 소스는 `monitor.fetch_html_generic(site)`로 **실제 제목을 눈으로 확인**하라. 메뉴("문화관광")·사진뉴스("폭우 피해현장")가 섞이면 그 셀렉터는 불량 → 활성화 보류하고 note에 사유 기록.
6. `python scripts/apply_enabled_from_diag.py` (드라이런) → `--write` — status=ok(실제 row 추출)인 imp_* 만 켜므로 "켰는데 0건" 함정이 구조적으로 차단된다.
7. 전체 테스트(`python -m pytest tests/ -q`) 후 커밋. 로컬에 `coverage_baseline.json`이 있으면
   `test_run_coverage_anomaly_check_no_network_no_real_file` 1건이 **원래부터 실패**한다(내 변경 탓 아님 — stash 대조로 확인).

## 알려진 사실 (2026-07-23 실측)
- 비활성 186개 중 **163개는 셀렉터 자체가 없다** — 병목은 "낡은 판정"이 아니라 "셀렉터 미배선".
- 인천 구·군청은 게시판 주소가 제각각. `eminwon` 패턴은 **부평 전용**(다른 구청은 404).
  확보된 주소: 연수 `/main/community/notify/notice.asp`, 남동 `/main/news/announce.jsp`.
- **0건의 원인은 4가지로 나뉜다 — 셀렉터만 의심하지 마라:**
  1. 셀렉터 미배선/오배선 → autofill + 눈 검증
  2. **기관 자체 소멸·개편** (예: 인천 중구청이 2026-07-01 영종구·제물포구로 분구되어 도메인 폐쇄) → 후속 기관 게시판으로 이관
  3. **URL 파라미터 체계 폐기** (예: 대전일자리경제진흥원 `menuId=` → `menuKey=`, 옛 URL은 HTTP 200인데 495바이트 에러페이지) → 200이라고 정상이 아니다, 본문 크기를 봐라
  4. **링크가 `javascript:void(0)`** → `fetch_html_generic`의 `bad_link` 분기에서 전건 skip돼 셀렉터를 고쳐도 0건 그대로.
     `selectors`에 `link_template` + (`link_id_attr` | `link_arg_re`) 합성 규칙을 넣어야 한다 (monitor.py:1729-1751).
- 서브도메인이 `event.`/`apply.` 인 소스는 접수시스템 껍데기일 수 있다 — 본사이트 공고 게시판을 찾아라.

- 통합포털(중소벤처24·보조금24 등) 비활성은 의도적일 수 있음(기업마당/K-Startup과 중복 수집 방지) — "고장"으로 단정하지 말 것.
